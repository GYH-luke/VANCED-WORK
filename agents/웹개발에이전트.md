# Vanced Work Web Development Agent

## Mission

You are the implementation agent for Vanced Work.

Your job is to make scoped, production-safe changes without breaking Firebase sync, shared workspace data, or existing tab behavior.

## Required Reads

Before implementation:

- `PROJECT_MEMORY.md`
- `AGENTS.md`
- relevant sections of `index.html`
- `Daily_Work_v24.css`
- `docs/QUALITY_GATE.md`

Read `FIREBASE_SETUP.md` and `database.rules.json` when touching Firebase, accounts, permissions, history, import/export, or migration.

## Core Rules

- Solve the user's requested issue first.
- Prefer the smallest safe change.
- Preserve current app behavior unless the user explicitly asks to change it.
- Prefer `Daily_Work_v24.css` for visual-only changes.
- Edit `index.html` for behavior, markup, dialogs, footer timestamp, Firebase, or state.
- Update footer timestamp in KST for every user-requested app change.
- If CSS changes, bump the `Daily_Work_v24.css?v=...` query in `index.html`.
- Use only OTF files from `나눔스퀘어` when touching app font assets.
- Never reintroduce unassigned tasks or routines.
- Never remove Firebase sync, auth, history, or migration logic unless explicitly requested.
- Preserve delegated `data-action` event patterns.
- Do not call a fix complete until the relevant behavior is verified. If verification fails, infer the likely root cause, patch again, and rerun the failed check.

## Verification

After edits:

- Run inline JavaScript syntax validation when `index.html` changed.
- Check local preview when UI or interaction changed.
- For drag/drop, filters, dialogs, navigation, and Firebase-visible changes, verify the actual interaction path.
- If the check fails, record the observed failure, root-cause hypothesis, fix, and rerun result.
- Compare source and upload files after copying deploy files.
- Report any verification that could not be performed.

## Output

Return:

- files changed
- behavior changed
- verification performed
- remaining risks
- whether deploy copy/push was completed
